# Real Deal - Dashboard Variants

## Goal

Turn the dashboard into a page that helps the user decide what matters fast.

## Current Baseline

- current live capture: [/tmp/realdeal-current.png](/tmp/realdeal-current.png)
- strong headline already exists
- the page has a lot of useful content
- the page still feels like stacked boards more than one composed surface

## Variant A - Thesis + Queue

### Idea

Big top thesis, one main work queue, everything else steps down.

### Layout

- top frame merges utility strip and hero
- left main column becomes the clear "do this now" area
- right column holds worlds, radar, and waiting
- recent movement and loose ends drop lower

### Pros

- strongest decision-first dashboard
- easiest to scan

### Cons

- some current modules get pushed down harder

## Variant B - Thesis + Split Rail

### Idea

Keep a strong thesis and use a right-side support rail for context.

### Layout

- top thesis band spans the page
- primary action area lives left and wide
- a consistent support rail on the right holds worlds, replies, radar, waiting
- movement and loose ends become lower sections

### Pros

- balanced
- closest to "Dia calm + Arc framing"
- best fit for the current content mix

### Cons

- needs discipline so the right rail does not become too busy

## Variant C - Report Stack

### Idea

Lean harder into a report feel with one strong opener and mostly vertical sections below.

### Layout

- top thesis is very strong
- most modules stack in one main column
- right side becomes lighter and more occasional

### Pros

- calmest
- cleanest reading flow

### Cons

- weaker sense of active command surface
- may underuse the app width

## Recommendation

Pick Variant B.

It keeps the dashboard useful while making it feel composed.

## Section Rules

- top strip and thesis should become one opening
- attention stays primary
- worlds becomes a support rail
- replies should stay high
- movement should stay visible, but lower
- loose ends should be present, but quieter
