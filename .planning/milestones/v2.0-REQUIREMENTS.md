# Requirements: Kinship Brain v2.0

**Defined:** 2026-03-29
**Source of Truth:** docs/Kinship Brain — MVP.pdf (Mar 28)
**Core Value:** Moj + team can import people, organize them into pods, see context and follow-ups, run pipelines and projects, pull reports, and share lists — without workarounds.

## V1 Scope (MUST SHIP)

### 1. Relationship Records (core data model)

- [x] **RR-01**: Contact + Company record types exist
- [x] **RR-02**: Record detail page shows pods/subpods, linked pipelines + projects, timeline
- [ ] **RR-03**: Multiple emails per contact + basic merge support
- [x] **RR-04**: List view with filters + bulk actions
- [x] **RR-05**: No separate deal or contact objects — relationship record is the only core object

### 2. Pods + Subpods (primary organization)

- [x] **POD-01**: Create unlimited pods + subpods
- [x] **POD-02**: Multi-pod assignment (one optional "primary")
- [x] **POD-03**: Pod-level required questions (enforced on add)
- [x] **POD-04**: Card + list views for pods (visual scan)
- [ ] **POD-05**: No artificial sub-pod cap (currently blocked at 3)

### 3. Pending Categorization Tray (intake flow)

- [x] **CAT-01**: Intake via manual add
- [x] **CAT-02**: Intake via CSV import
- [ ] **CAT-03**: Intake via basic email parsing
- [x] **CAT-04**: Categorization modal — assign pods/subpods, complete required questions, add notes + referral source
- [x] **CAT-05**: Saves directly into records + timeline

### 4. Pipelines (foundational Kanban)

- [ ] **PIPE-01**: Create pipelines with custom stages
- [ ] **PIPE-02**: Opportunity cards linked to relationship records
- [ ] **PIPE-03**: Move cards across stages
- [ ] **PIPE-04**: Pipeline-specific fields (visible in pipeline context, accessible for reporting)
- [ ] **PIPE-05**: Pipelines can be hidden (not deleted)
- [ ] **PIPE-06**: Open relationship from pipeline, open pipeline from relationship

### 5. Projects (foundational module)

- [ ] **PROJ-01**: Create projects as containers (events, outreach, initiatives)
- [ ] **PROJ-02**: Each project supports attached contacts, attached opportunities, notes/updates
- [ ] **PROJ-03**: Simple project overview (no advanced dashboards yet)

### 6. Basic Enrichment (controlled + minimal)

- [ ] **ENRICH-01**: Email metadata (last interaction + subject preview)
- [ ] **ENRICH-02**: Enrichment via web search writes to timeline
- [ ] **ENRICH-03**: Enrichment opt-in per contact or pod (tracked contacts)

### 7. Reporting (V1 = essential reports)

- [ ] **RPT-01**: Pod distribution report
- [ ] **RPT-02**: Pipeline conversion / velocity report
- [ ] **RPT-03**: Engagement / touchpoint activity report
- [ ] **RPT-04**: Export CSV (PDF optional if low effort)
- [ ] **RPT-05**: Save favorite reports

### 8. Nurturing / Maintenance Hub

- [ ] **NURT-01**: Single maintenance feed showing stale relationships, upcoming milestones, missing required info
- [ ] **NURT-02**: Simple actions: add note, create follow-up, snooze / mark done
- [ ] **NURT-03**: Follow-ups live in dashboard layers, not as tasks
- [ ] **NURT-04**: Pod-based cadence drives default follow-up behavior
- [ ] **NURT-05**: Record-level override configurable

### 9. Sharing + Exports

- [ ] **SHARE-01**: Export any filtered list (pods, subpods, reports)
- [ ] **SHARE-02**: Generate read-only share links for curated lists (LP lists, referrals, city guides/vendors)

### Cross-Cutting (from MVP Checklist)

- [x] **XCUT-01**: Relationship record is the only core object
- [x] **XCUT-02**: Pipelines/campaigns reference relationships (no copying / no new instances)
- [x] **XCUT-03**: Cross-module navigation with zero context loss
- [x] **XCUT-04**: Single timeline per relationship — all connections logged
- [ ] **XCUT-05**: No duplicate records anywhere (non-negotiable guardrail)
- [ ] **XCUT-06**: No silent overwrites (non-negotiable guardrail)
- [ ] **XCUT-07**: Everything traceable and explainable (non-negotiable guardrail)
