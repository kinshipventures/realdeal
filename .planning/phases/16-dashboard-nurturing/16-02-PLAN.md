**1. Create `src/components/nurturing/NurturingRow.tsx`**

Single contact row used throughout the nurturing hub. Three actions per D-11.

Props:
```typescript
interface NurturingRowProps {
  contact: Contact
  signal: string        // e.g. "Overdue by 12 days", "No contact in 94 days", "Birthday in 3 days", "Missing: Industry, Stage"
  signalColor: string   // semantic color for the signal badge
  onSnooze: (id: string) => void
  onInteractionLogged: () => void
}
```

Structure:
- Left: `<Avatar name={contact.name} size={32} />` + contact name (fontSize 14, fontWeight 500) + signal text below (fontSize 12, color = signalColor)
- Right: three action buttons in a row:
  1. **Log interaction**: A button that toggles open a horizontal pill row of type buttons (call, email, text, meeting, note). Use `TYPE_ICONS` from InteractionSection.tsx for the icon set. On type click, call `logInteraction(contact.id, type, '')` from airtable.ts, then call `onInteractionLogged()`. The pill row appears inline below the contact row (not a modal). Each pill: 28px height, icon + type label, rounded, muted background. Active state on click.
  2. **Snooze 30d**: Button with clock/snooze icon (use lucide `Clock` or `BellOff`). On click: call `onSnooze(contact.id)`. No confirmation needed.
  3. **Navigate**: The entire row body (name area) is clickable — `navigate(`/contact/${contact.id}`)` using `useNavigate()` from react-router. Note: the route is `/contact/:id` NOT `/record/:id`.

Style: Row has `padding: '12px 20px'`, bottom border `1px solid rgba(0,0,0,0.06)`. Hover: subtle background `rgba(0,0,0,0.02)`. Action buttons are ghost-style (transparent background, icon only, 32px tap target). The log interaction and snooze buttons stop propagation so they don't trigger row navigation.

Language per CONTEXT.md specifics: follow-ups should feel "instructional, not assigned" — signal text should be "Consider reaching out" not "Overdue: contact now." For overdue: "Last reached out {N} days ago". For dormant: "No contact in {N} days". For birthdays: "Birthday in {N} days" or "Birthday today". For missing fields: "Missing: {field1}, {field2}".

**2. Create `src/components/nurturing/NurturingHub.tsx`**

Route component for `/pulse/nurturing`. Fetches its own data (separate from Dashboard.tsx — this is a standalone route).

Data fetching (in a single `useEffect`):
- `getContacts()` → contacts
- `getPods()` → pods
- `getAllInteractions()` → interactions
- `getFieldConfigs()` → fieldConfigs (for NURT-04 missing fields detection)
- `getSnoozedIds()` → snoozedIds (initial state)

URL filter param: Read `?filter=focus|overdue|stale|dates|hygiene` from `useSearchParams()`. Use this to auto-scroll to the relevant section on mount via `useEffect` + `scrollIntoView()` on section ref. Do NOT filter out other sections — all sections always render, but the matching section scrolls into view.

Computed data (via `useMemo`):

- **dueSoonContacts**: Same logic as Dashboard.tsx `overdueContacts` — for each contact, check `isOverdue(contact, cadence)` per-pod. Exclude `isInGracePeriod(contact)`. Exclude snoozed. Sort by overdue severity descending.

- **staleContacts**: `contacts.filter(c => isDormant(c) && !snoozedIds.has(c.id) && !isInGracePeriod(c))`. Map to `{ contact, days: daysSinceContact(c) }`. Sort by days descending.

- **upcomingDates**: `getUpcomingBirthdays(contacts, pods)` + follow-up items (contacts with `next_follow_up_date` within 14 days). Merge and sort by daysUntil ascending.

- **dataHygieneItems**: Two sub-categories:
  1. **Missing required fields**: For each contact, for each fieldConfig where `required === true` AND `contact.list_ids.includes(fieldConfig.pod_id)`, check if the contact has a value for that field (check `contact[fieldConfig.field_name]` or contact's custom field values). Collect contacts with any missing required field. Group by contact, list missing field names.
  2. **Pods at capacity**: For each pod with a `capacity` value, compare `contacts.filter(c => c.list_ids.includes(pod.id)).length` against `pod.capacity`. If at or over, include `{ pod, currentCount, capacity }`. Do NOT list individual contacts to manage out — let user click through to pod detail per Research recommendation.
