**Key implementation patterns:**

1. **getPods()** - `supabase.from('pods').select('*').order('created_at')`. No mapper needed - snake_case columns match Pod interface directly. Demo mode check first.

2. **getContacts(categoryId?)** - THE critical function. Must use embedded selects for junction tables:
```typescript
let query = supabase.from('contacts').select(`
  *,
  contact_pods(pod_id, is_primary),
  contact_categories(category_id)
`)
```
Then flatten junction data to match Contact interface:
```typescript
return (data ?? []).map(row => ({
  ...row,
  list_ids: (row.contact_pods ?? []).map((cp: any) => cp.pod_id),
  category_ids: (row.contact_categories ?? []).map((cc: any) => cc.category_id),
  primary_list_id: (row.contact_pods ?? []).find((cp: any) => cp.is_primary)?.pod_id ?? null,
}))
```
If `categoryId` is provided, filter client-side (PostgREST embedded filter may not work reliably per RESEARCH open question #2): fetch all contacts, then filter to those whose `category_ids` includes `categoryId`.

3. **createContact(data)** - Insert into `contacts` table, then insert into `contact_pods` and `contact_categories` junction tables using `data.list_ids` and `data.category_ids`. Handle `primary_list_id` by setting `is_primary: true` on the matching `contact_pods` row. Return the full Contact object with flattened arrays.

4. **updateContact(id, data)** - Update `contacts` row. If `data.list_ids` is present, delete existing `contact_pods` rows for this contact and re-insert. Same for `data.category_ids` -> `contact_categories`. If `data.primary_list_id` changed, update `contact_pods.is_primary`. Invalidate contacts cache after.

5. **deleteContact(id)** - `supabase.from('contacts').delete().eq('id', id)`. CASCADE handles junction cleanup. Invalidate cache.

6. **getAllInteractions()** - `supabase.from('interactions').select('*').order('date', { ascending: false })`. No junction flattening needed.

7. **getInteractions(contactId)** - REMOVE the `rec[A-Za-z0-9]{14}` validation (per RESEARCH Pitfall 3). Just filter: `.eq('contact_id', contactId)`.

8. **createInteraction(data)** - Direct insert. No linked record resolution needed - IDs are already UUIDs.

9. **logInteraction(contactId, data)** - Same logic as current: call createInteraction, then updateContact last_contacted_at if type is not 'note'. Keep optimistic cache append.

10. **isOverdue(contact, cadence)** and **isInGracePeriod(contact)** - Pure functions, no data access. Copy directly from airtable.ts unchanged.

11. **getOverdueContacts(podId, cadence)** - Fetch contacts, filter by pod membership and overdue status. Same logic, different data source.

12. **getCampaigns()** - Fetch campaigns with embedded `campaign_contacts(contact_id)` to populate `contact_ids` array:
```typescript
const { data } = await supabase.from('campaigns').select(`*, campaign_contacts(contact_id)`)
return data.map(row => ({
  ...row,
  contact_ids: (row.campaign_contacts ?? []).map((cc: any) => cc.contact_id),
}))
```

13. **getCampaignContacts(campaignId)** - `supabase.from('campaign_contacts').select('*').eq('campaign_id', campaignId)`

14. **createCampaign, addContactToCampaign, updateCampaignContactStatus, completeCampaign** - Direct Supabase CRUD. No linked record arrays to manage (campaign_contacts is already a junction table in both systems).

15. **getPipelines, createPipeline, updatePipeline** - Direct CRUD on `pipelines` table.

16. **getPipelineStages(pipelineId?)** - Fetch all, filter client-side if pipelineId provided (matches current pattern).

17. **getOpportunities()** - Fetch with embedded `opportunity_contacts(contact_id)` to populate `relationship_ids`:
```typescript
const { data } = await supabase.from('opportunities').select(`*, opportunity_contacts(contact_id)`)
return data.map(row => ({
  ...row,
  relationship_ids: (row.opportunity_contacts ?? []).map((oc: any) => oc.contact_id),
}))
```

18. **createOpportunity(name, stageId, relationshipIds)** - Insert opportunity, then insert `opportunity_contacts` junction rows.

19. **updateOpportunity** - Update opportunity row. If relationship_ids would change, handle junction table update (check current airtable.ts to see if this is supported - it currently only updates name/stage_id/notes/priority/status, not relationship_ids, so just do direct update).

20. **getProjects()** - Fetch with embedded `project_contacts(contact_id)` and `project_opportunities(opportunity_id)`:
```typescript
const { data } = await supabase.from('projects').select(`*, project_contacts(contact_id), project_opportunities(opportunity_id)`)
return data.map(row => ({
  ...row,
  relationship_ids: (row.project_contacts ?? []).map((pc: any) => pc.contact_id),
  opportunity_ids: (row.project_opportunities ?? []).map((po: any) => po.opportunity_id),
}))
```

21. **addRecordToProject(projectId, recordId)** - Insert into `project_contacts`. Return updated project.

22. **removeRecordFromProject(projectId, recordId)** - Delete from `project_contacts`. Return updated project.

23. **addOpportunityToProject / removeOpportunityFromProject** - Same pattern with `project_opportunities`.

24. **addProjectNote(projectId, note)** - Fetch current notes, append new note, update. Same logic as current.

25. **mergeRecords(primaryId, secondaryId, mergedData, user)** - Complex function. Read current airtable.ts implementation carefully. Must:
    - Update primary contact with merged data
    - Move secondary's interactions to primary (update contact_id)
    - Move secondary's junction rows (contact_pods, contact_categories, opportunity_contacts, project_contacts, campaign_contacts) to primary - handle duplicates by skipping if already exists
    - Log a merge_event interaction
    - Delete secondary contact
    - Invalidate all caches

26. **getContactsByType(type)** - `getContacts()` then filter by `type`.

27. **getActiveContacts()** - `getContacts()` then filter by `status === 'Active'`.

28. **getPendingContacts()** - `getContacts()` then filter by `status === 'Pending'`.
