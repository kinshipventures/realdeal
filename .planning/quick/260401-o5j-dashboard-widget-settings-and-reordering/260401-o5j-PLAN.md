**WrappedCard.tsx:**

**useDashboardConfig.ts:**
- Add `order: WidgetId[]` to `DashboardConfig` and `StoredConfig`
- Define `DEFAULT_ORDER: WidgetId[]` matching current render order: `['pending-tray', 'pod-health', 'wrapped', 'todays-focus', 'coming-up', 'needs-attention', 'recent-activity', 'quick-links']` (equity excluded - it lives in the header band, not the widget flow)

- Update `saveConfig()` to persist `order` array
- Add `reorderWidgets(fromIndex: number, toIndex: number)` callback that splices the order array and saves. Standard array move: remove item at fromIndex, insert at toIndex.
- Bump storage key to `realdeal:dashboard-config:v2` so old configs without order get fresh defaults
  </action>

<task type="auto">
  <name>Task 2: Add drag-to-reorder in DashboardSettings and render Dashboard in order</name>
  <files>src/components/dashboard/DashboardSettings.tsx, src/components/dashboard/Dashboard.tsx</files>
  <action>
**DashboardSettings.tsx:**
- Accept `onReorder: (from: number, to: number) => void` prop and `order` from config
- Replace the static `ALL_WIDGETS.map()` list with `config.order.map()` (filtered to show all widgets, visible or not, in the config's order)
- Add native HTML drag-and-drop to each widget row:
  - 6-dot drag handle (3 rows of 2 dots) on the left side of each row, color `var(--color-text-tertiary)`
  - `draggable` on the row, `onDragStart` sets the dragged index
  - `onDragOver` (preventDefault to allow drop) with a visual indicator line (2px solid `var(--color-brand)`) above or below the hovered row
  - `onDrop` calls `onReorder(fromIndex, toIndex)`
  - `onDragEnd` clears state
- Keep toggle switches on the right side as-is
- Use component state for drag tracking (`dragIndex`, `overIndex`). No external library.

**Dashboard.tsx:**
- Pass `config.order` and `reorderWidgets` to DashboardSettings (add to existing props spread)
- Refactor the widget rendering in the "Rest of dashboard" section to be order-driven:
  - Create a `WIDGET_REGISTRY` map inside Dashboard that maps each `WidgetId` to a render function returning JSX (or null if data not ready)
  - Iterate `config.order.filter(id => isVisible(id))` and render each widget from the registry
  - Group into sections: widgets are rendered in order but still grouped under their section headings. Use the existing section groupings:
    - "network pulse": `pod-health`, `wrapped`
    - "action items": `todays-focus`, `coming-up`, `needs-attention`
    - "activity & links": `recent-activity`, `quick-links`
    - standalone (no heading): `pending-tray`
  - Within each section, render widgets in the order they appear in `config.order`
  - A section heading only renders if at least one of its widgets is visible and in the order
  - The section order itself follows the first appearance of any of its widgets in `config.order`
  </action>
